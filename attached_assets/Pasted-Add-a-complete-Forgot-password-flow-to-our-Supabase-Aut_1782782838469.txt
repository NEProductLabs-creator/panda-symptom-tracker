Add a complete "Forgot password?" flow to our Supabase Auth signin. Currently there is no way for users to reset their password.

Requirements:

1. SIGN-IN SCREEN: On the existing sign-in form (in Landing.tsx or wherever the email/password fields live), add a "Forgot password?" text link directly below the password field, right-aligned, in a muted color. Tapping it opens the reset request flow.

2. RESET REQUEST SCREEN: New route /auth/forgot-password. Single email input plus a "Send reset link" button. On submit, call:
   supabase.auth.resetPasswordForEmail(email, {
     redirectTo: `${window.location.origin}/auth/reset-password`
   })
   Always show the same success message regardless of whether the email exists ("If an account exists for that email, we just sent a reset link. Check your inbox and spam folder."). Do not leak whether the email is registered. Add a 30-second cooldown on the submit button.

3. RESET COMPLETE SCREEN: New route /auth/reset-password. When Supabase redirects the user here from the email link, they arrive with a recovery session active. Detect this by listening for the PASSWORD_RECOVERY event on supabase.auth.onAuthStateChange. Show two fields: new password, confirm new password. Validate min 8 chars and match. On submit:
   supabase.auth.updateUser({ password: newPassword })
   On success, show "Password updated. You're signed in." and redirect to the dashboard. On error, show the message.

4. SUPABASE CONFIG: Make sure /auth/reset-password is added to the Redirect Allowlist in Supabase dashboard (Authentication, URL Configuration). Also update the "Reset Password" email template to match our brand voice: greeting "Hi there," sender name "PANS/PANDAS Companion," friendly supportive tone, clear single CTA button using the {{ .ConfirmationURL }} token, plain-text fallback included.

5. ERROR HANDLING: If the user lands on /auth/reset-password without an active recovery session (link expired, link reused), show: "This reset link has expired or was already used. Request a new one." with a button back to /auth/forgot-password.

6. ACCESSIBILITY: All inputs labeled. Buttons reach minimum 44px tap target. Error and success messages use role="status" or role="alert" appropriately.

7. DEMO MODE: This entire flow should be a no-op in demo mode. If a user is in demo mode and somehow lands on these routes, redirect them back to the demo dashboard with a toast: "Password reset is disabled in demo mode."

8. DEMO DATA: Per project convention, add a one-time scripted demo of the forgot-password screen to the demo mode showcase if there's an onboarding tour or feature gallery, so demo users can see the flow exists. If no such gallery exists, skip this step.

Test the full loop end to end with a real email before considering this done: sign in, click forgot password, submit email, click the link in the email, set new password, confirm you land in the dashboard signed in.
